# خطة بناء داش بورد متكاملة (ادمن + عميل)

سأبني نظام إدارة كامل يغطي كل ما طلبت. العمل كبير لذا سأنظمه في 4 مراحل واضحة.

---

## 1) قاعدة البيانات (مايجريشن واحد)

### تفعيل تخزين الصور
- إنشاء bucket عام اسمه `media` (للصور: أحواض، مقالات، خدمات، صفحات).
- سياسات: أي زائر يقرأ، الادمن فقط يرفع/يحذف.

### جداول جديدة

- **services** — محتوى صفحة خدماتنا (عنوان، وصف، أيقونة، صورة، ترتيب، منشور).
- **site_pages** — محتوى الصفحات القابلة للتعديل (about, home_hero, business_solutions…): مفتاح، عنوان، محتوى JSON مرن، صورة.
- **customer_tanks** — أحواض العميل (اسم، نوع، مقاس، حجم لتر، سمك/نباتات، تاريخ التركيب، صورة، ملاحظات، `user_id`).
- **maintenance_reports** — تقارير صيانة لكل حوض (تاريخ الزيارة، الفني، ملاحظات، الإجراءات المنفذة، الحالة العامة).
- **water_tests** — قراءات تحليل الماء (pH, NH3, NO2, NO3, KH, GH, TDS, درجة الحرارة، الملوحة، تاريخ).
- **appointments** — مواعيد العميل (نوع: استشارة/معاينة/تصميم/صيانة، تاريخ مقترح، حالة، ملاحظات الادمن).

### تعديلات

- إضافة عمود `image_path` (مسار في storage) إلى `projects` و `articles`.
- إضافة دور `staff` إلى `app_role` (admin/staff/customer).
- دالة `has_any_role` للتحقق من دور أو أكثر.

### RLS باختصار
- الادمن: قراءة/كتابة كل شيء.
- الموظف (staff): قراءة الطلبات والعملاء + تحديث حالة الطلبات/التقارير (بدون حذف المستخدمين أو الأدوار).
- العميل: يقرأ/يضيف/يعدل أحواضه ومواعيده فقط؛ يقرأ تقارير الصيانة والتحاليل الخاصة بأحواضه فقط.
- المحتوى العام (services, site_pages, projects, articles, testimonials المنشورة): قراءة عامة.

### نقل البيانات الحالية
- نقل الأحواض من `src/data/projects.ts` إلى جدول `projects` (seed داخل المايجريشن).
- نقل التقييمات المكتوبة سابقاً (Sara, Danah…) إلى `testimonials`.

## 2) لوحة الادمن `/admin` — متكاملة

شريط جانبي ثابت يحتوي:

- **نظرة عامة** — إحصائيات لحظية.
- **الطلبات** — تواصل/استشارات/مواعيد (موجود + تطوير).
- **العملاء** — كل المسجلين، عرض أحواضهم وطلباتهم، تغيير الدور (admin/staff/customer)، حذف.
- **الموظفين** — إضافة/حذف موظف، تعيين صلاحيات.
- **محتوى الصفحات** قسم كبير:
  - **اعمالنا (Projects)** — قائمة كل الأحواض، تعديل أي حقل، رفع صور متعددة، تحديد الصورة الرئيسية، حذف، إضافة جديد.
  - **خدماتنا (Services)** — CRUD مع رفع صور وأيقونات.
  - **حلول الأعمال** — تعديل أقسام الكوفيهات/المطاعم/الفعاليات…
  - **الصفحة الرئيسية** — تعديل الـ hero والأرقام والنصوص.
  - **من نحن** — تعديل النص والصور.
  - **المقالات** — موجود + رفع صورة الغلاف.
  - **التقييمات** (بدل "الشهادات") — CRUD كامل، استيراد الموجود مسبقاً.
- **الإعدادات** — معلومات التواصل، واتساب، روابط السوشيال.

### رفع الصور
مكوّن `<ImageUploader />` موحّد: زر اختيار من الجهاز → رفع إلى bucket `media` → حفظ المسار في DB → عرض المعاينة → إمكانية الحذف والاستبدال. للأحواض: رفع متعدد + سحب لإعادة الترتيب + اختيار الصورة الرئيسية.

## 3) لوحة العميل `/account` — داش بورد منفصل

شريط جانبي بأقسام احترافية:

- **نظرة عامة** — ترحيب + ملخص (عدد الأحواض، آخر صيانة، الموعد القادم).
- **أحواضي** — قائمة أحواض العميل، إضافة حوض جديد (اسم، مقاس، نوع، صورة، تاريخ التركيب).
- **تفاصيل الحوض** (نقرة على حوض) — البيانات + سجل تقارير الصيانة + جدول قراءات الماء + رسم بياني بسيط لاتجاه pH/NO3.
- **مواعيدي** — كل المواعيد المحجوزة وحالتها.
- **طلباتي** — كل طلبات التواصل والاستشارات السابقة.
- **اطلب خدمة** — أزرار سريعة: طلب استشارة / طلب معاينة / طلب تصميم / طلب صيانة (لحوض موجود).
- **ملفي الشخصي** — الاسم، الجوال، تغيير كلمة المرور.

---

## 4) ربط الموقع العام بقاعدة البيانات

- صفحة `/portfolio` تقرأ من جدول `projects` (بدل الملف الثابت).
- صفحة `/services` تقرأ من `services`.
- الصفحة الرئيسية والتقييمات تقرأ من DB.
- المقالات تقرأ من DB.

---

## ملاحظات تقنية

- كل الكتابة عبر `supabase` المباشر مع RLS (لا حاجة لـ server functions إضافية لمعظم العمليات).
- استخدام `react-query` للتخزين المؤقت في الادمن.
- رفع الصور: `supabase.storage.from('media').upload(...)` + حفظ المسار، والقراءة عبر `getPublicUrl`.
- الحفاظ على التصميم الحالي (Glassmorphism + ذهبي على بحري داكن، RTL، خط IBM Plex Sans Arabic).

## ما خارج هذه الخطة (لاحقاً إن رغبت)
- إشعارات بريد/واتساب تلقائية للموظفين عند طلب جديد.
- نظام محادثة داخلي بين العميل والفني.
- فوترة واشتراكات شهرية.
